# Claude Skill Porter — документация на русском

## Что это вообще такое

`claude-skill-porter` — это skill для Codex, который помогает переносить skills из Claude Code в формат, пригодный для Codex.

Простыми словами:

- у тебя есть skill, написанный под Claude Code
- ты хочешь использовать его в Codex
- этот инструмент помогает перенести то, что переносится нормально
- и отдельно честно показывает, что не перенеслось автоматически и где есть риск

Это важно, потому что Claude Code и Codex похожи не полностью. Если просто копировать файлы "как есть", можно получить красивый, но сломанный результат.

## Зачем он нужен

Этот skill нужен, когда:

- в проекте есть папка `.claude/skills/...`
- в проекте есть старые `.claude/commands/*.md`
- команда хочет переехать с Claude Code на Codex
- нужно сначала понять риски миграции, а не просто слепо копировать

## Главная идея

Мы не пытаемся сделать магическую "полную совместимость".

Вместо этого делаем честный и безопасный перенос:

1. переносим то, что реально переносится
2. отдельно отмечаем всё спорное
3. оставляем понятный отчёт, что нужно доделать руками

Это сильно лучше, чем "успешная миграция", после которой skill внезапно ведёт себя уже не так, как раньше.

## Что именно создаёт этот skill

На выходе должен появляться новый skill для Codex, внутри которого есть:

- `SKILL.md`
- `agents/openai.yaml`
- `references/migration-report.md`
- `references/original-frontmatter.md`

Дополнительно копируются:

- `scripts/`
- `references/`
- `assets/`

если они были у исходного Claude skill.

## Что переносится хорошо

Вот что обычно можно переносить автоматически:

- имя skill
- описание skill
- основной markdown-текст из `SKILL.md`
- папки `scripts/`, `references/`, `assets/`
- простой `argument-hint`
- простой `allowed-tools`, если там просто список инструментов без сложной логики

Это и есть "зелёная зона" миграции.

## Где начинаются сложности

Проблемы начинаются там, где Claude Code добавляет своё специальное поведение.

Особенно рискованные вещи:

- `context: fork`
- `agent`
- `hooks`
- `paths`
- `model`
- `effort`
- `disable-model-invocation`
- `user-invocable`
- конструкции вида `$ARGUMENTS`
- переменные вида `${CLAUDE_*}`
- вставка shell-команд через `!`

Почему это важно:

- это не просто текст
- это поведение платформы
- у Codex такого поведения может не быть вообще или оно может работать иначе

Поэтому такие штуки нельзя честно назвать "автоматически перенесёнными".

## Что делает конвертер

Скрипт `scripts/convert_claude_skill.py` делает базовую и безопасную часть работы:

1. принимает путь к Claude skill или legacy command
2. читает frontmatter и основной markdown
3. создаёт новую skill-папку под Codex
4. копирует переносимые поля
5. копирует папки `scripts/`, `references/`, `assets/`
6. пишет отчёт о рисках и несовместимостях

Самое важное:

если он видит опасные или неполностью совместимые вещи, он не скрывает это, а пишет в отчёт.

Дополнительно теперь есть:

- `migration-manifest.json` — машинно-читаемый файл для автоматических проверок
- режим `--strict` — если есть опасные несовместимости, команда завершится с ошибкой
- safe auto-rewrite типовых конструкций вроде `$ARGUMENTS`, простых `${CLAUDE_*}` и простых `!`-вставок
- batch-режим для миграции всего репозитория или всей `.claude`-папки за один запуск

## Что такое migration report

`migration-report.md` — это главный защитный механизм.

Он отвечает на 4 вопроса:

- что скопировалось автоматически
- что перенеслось только частично
- что требует ручной доработки
- почему это рискованно

Это нужно для того, чтобы после миграции никто не думал: "ну раз папка создалась, значит всё уже работает".

## Что такое migration manifest

`migration-manifest.json` — это такой же отчёт, но в формате для машин.

Он нужен, если позже ты захочешь:

- проверять качество миграции автоматически
- останавливать публикацию при рисках
- подключать GitHub Actions или другой CI

То есть markdown нужен человеку, а JSON нужен автоматике.

## Что такое batch-режим

Batch-режим нужен, когда ты хочешь переносить не один skill, а весь проект сразу.

Тогда конвертер:

- ищет `.claude/skills/*`
- ищет `.claude/commands/*.md`
- переносит всё найденное
- создаёт общий `batch-migration-report.md`
- создаёт общий `batch-migration-manifest.json`

Это уже ближе к нормальному продукту, а не к одноразовому скрипту.

## Самые важные риски простым языком

### 1. Ложное ощущение, что всё совместимо

Самый опасный риск.

Файлы могут красиво скопироваться, но реальное поведение skill уже будет другим.

### 2. Пропадает скрытая логика Claude

Если исходный skill использовал `hooks`, `context: fork` или shell-injection, то после миграции эта скрытая магия может исчезнуть.

### 3. Ломаются аргументы

Если skill рассчитывал на `$ARGUMENTS`, после переноса это может перестать работать или стать непонятным.

### 4. Неправильно переносятся права на инструменты

`allowed-tools` может выглядеть похоже, но смысл и правила в двух системах могут отличаться.

### 5. Портируются приватные или внутренние данные

В `references/` или `scripts/` могут лежать внутренние ссылки, токены, служебные инструкции. Перед публикацией на GitHub это обязательно надо просмотреть.

### 6. Сообщество скажет: "Почему публикация не падает, если skill ещё кривой?"

Это справедливая критика.

Поэтому я добавил `--strict`: если остаются ручные проблемы, команда завершается с ошибкой. Это полезно именно перед публикацией.

### 7. Сообщество скажет: "Где формат для автоматической проверки, а не только markdown?"

Тоже справедливо.

Поэтому теперь кроме markdown-отчёта создаётся ещё и `migration-manifest.json`.

### 8. Сообщество скажет: "Зачем это вообще, если можно просто остаться на Claude Code?"

Это тоже нормальный вопрос.

Честный ответ такой:

- если переезд в Codex не нужен, то да, можно не запариваться
- этот проект нужен не всем подряд
- он нужен именно тогда, когда есть цель переезда, переиспользования skill'ов или публикации Codex-версии

То есть смысл проекта не в том, чтобы "победить Claude Code", а в том, чтобы сделать миграцию в Codex безопасной, если она уже нужна.

## Почему документация вынесена отдельно

Я специально не запихивал все объяснения в основной `SKILL.md`, потому что:

- основной skill должен быть коротким и рабочим
- подробности и риски лучше читать отдельно
- так проще сначала понять идею человеком, а потом уже использовать сам skill

До GitHub-публикации нужно проверить:

- что skill валиден
- что конвертер реально работает на тестовом примере
- что в документации нет лишнего мусора
- что в `references/` и `scripts/` нет приватных данных
- что описание skill понятно будущему пользователю
- что `--strict` отрабатывает так, как мы ожидаем

## Короткий вывод

Этот проект стоит делать не как "волшебный переносчик", а как "честный переносчик с отчётом о рисках".

Именно это даёт доверие:

- безопасно переносит простые случаи
- не врёт про сложные случаи
- помогает понять, что ещё нужно доделать руками
