# Claude Skill Porter — чеклист до v1

## Что значит v1 для этого проекта

Для этого проекта `v1` — это не "идеальная совместимость с Claude Code".

Для нас `v1` значит:

- инструмент решает частые реальные кейсы
- ограничения честно описаны
- результат можно проверить тестами
- проект можно показать людям на GitHub без ощущения "сырого черновика"

## Чеклист v1

### 1. Усилить тесты

Статус:

- сделано

Что нужно:

- добавить fixture-based тесты
- добавить больше реальных примеров Claude skills
- проверить итоговые артефакты не только по статусу, но и по содержимому

Минимальный набор фикстур:

- простой Claude skill
- legacy command
- skill с `context: fork`
- skill с `hooks`
- skill с `${CLAUDE_*}`
- skill с несколькими rewrite-паттернами сразу

Критерий готовности:

- после изменения конвертера мы быстро видим, что сломалось

### 2. Добавить golden outputs

Статус:

- сделано

Что нужно:

- сохранить ожидаемые результаты для фикстур
- сравнивать реальный вывод с эталонным

Зачем:

- чтобы изменения в конвертере не ломали старые кейсы незаметно

Критерий готовности:

- для ключевых сценариев есть эталонный ожидаемый результат

### 3. Сделать нормальный README

Статус:

- сделано

Что нужно включить:

- что это за проект
- какую проблему он решает
- что умеет
- что не умеет
- быстрый запуск
- пример входа
- пример выхода
- ограничения
- честное объяснение, что это не full compatibility layer

Критерий готовности:

- новый человек понимает проект за 2-3 минуты

### 4. Сделать release checklist

Статус:

- сделано

Перед публикацией должно быть явно записано:

- skill validation
- unit tests
- smoke test
- strict run
- privacy review
- финальная проверка docs

Критерий готовности:

- релиз не держится только на памяти

### 5. Сделать минимальный CI

Что нужно:

- запуск тестов
- проверка базовой исправности конвертера

Статус:

- сделано

Примечание:

- локальную `skill validation` мы оставляем в release checklist, потому что GitHub CI не может опираться на локальный путь к Codex validator

Критерий готовности:

- любой коммит можно автоматически проверить

### 6. Сделать repo cleanup

Статус:

- сделано

Что нужно:

- убрать `.DS_Store`
- проверить структуру проекта
- добавить `.gitignore`
- добавить `LICENSE`

Опционально:

- добавить простой `Makefile` или команды запуска

Критерий готовности:

- репозиторий выглядит аккуратно и предсказуемо

### 7. Зафиксировать non-goals

Статус:

- сделано

Нужно прямо написать, что проект:

- не эмулирует Claude hooks
- не гарантирует полную parity
- не переносит все сложные случаи автоматически

Критерий готовности:

- у сообщества не возникает ложных ожиданий

## Что уже можно считать готовым

У нас уже есть:

- single migration
- batch migration
- safe auto-rewrite типовых случаев
- strict mode
- manifests и reports
- fixture-based автотесты
- golden outputs
- README
- release checklist
- минимальный CI
- `.gitignore` и `LICENSE`

Это значит:

- проект уже выше MVP
- проект доведён до узкого `v1` для честной GitHub-публикации

## Порядок работ до v1

Лучший порядок такой:

1. fixture-based тесты
2. golden outputs
3. README
4. release checklist
5. CI
6. repo cleanup
7. финальная GitHub-подготовка

На текущем этапе пункты 1-6 закрыты.

## Чего не надо брать в v1

Вот это можно оставить на потом:

- полную поддержку hooks
- полную поддержку subagents
- 100% совместимость Claude Code -> Codex
- сверхумные rewrite для любых сложных кейсов

Иначе проект снова начнёт расползаться.

## Критерий "можно назвать v1"

Проект можно считать `v1-ready`, когда:

- человек с GitHub быстро понимает, что это такое
- есть понятный quick start
- тесты покрывают основные сценарии
- CI подтверждает базовую исправность
- ограничения явно описаны
- публикация проходит по checklist, а не "на глаз"

Текущий статус:

- эти критерии закрыты для текущего узкого scope проекта
